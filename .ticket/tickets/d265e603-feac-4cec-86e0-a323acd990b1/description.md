# Plan: Search Algorithm Event Emission Refactoring

**Date:** 2026-03-03
**Interview:** `agents/interviews/20260303_SEARCH_EVENT_REFACTORING.md`

## Objective

Refactor the search algorithm's event emission system to eliminate duplication, use `PathNode` instead of raw `usize`, implement trait-based event derivation, add a two-phase tentative/confirmed root model, populate `EdgeRef` with real data, and create a separate query path event stream — including all frontend updates.

## Analysis

### Current State

Event emission is scattered across `SearchState::next()`, `emit_compare_events()`, and `finish_root_cursor()` with:
- `PoppedNode` extracting fields that `SearchNode` should provide
- `CompareInfo`/`PrefixChildInfo` using raw `usize` instead of `PathNode`
- Duplicated pop logic across `pop_node`+`process_node` and `pop_and_process_one`
- ~6 repeated `EdgeRef { from, to, pattern_idx: 0, sub_index: 0 }` placeholder constructions
- ~150 lines of manual child token extraction + event emission in `finish_root_cursor`
- No query-side events
- Rejected parent candidates briefly shown as confirmed root

### Desired State

- `SearchNode` provides `root_token()`, `is_parent()` methods directly
- `CompareInfo` and `PrefixChildInfo` use `PathNode`
- Single `pop_and_process_one` API with pre/post hooks for event injection
- `IntoTransition` trait implemented for result types
- `Transition` variants use `PathNode` instead of split usize fields
- `EdgeRef` populated with real `ChildState::root_child_index()` data
- `VizPathGraph` has `root_tentative: bool` — `VisitParent` sets tentative, `CandidateMatch` confirms
- Separate query path event stream with its own `path_id` and `VizPathGraph`
- `RootCursor::advance_to_next_match` accepts event collector

### Key Changes Required

1. Add methods to `SearchNode` (`root_token()`, `is_parent()`)
2. Refactor `CompareInfo`/`PrefixChildInfo` to use `PathNode`
3. Remove `PoppedNode`, `pop_node`, `process_node` — keep `pop_and_process_one` with hooks
4. Change `Transition` variants to use `PathNode`
5. Add `VizPathGraph.root_tentative` and two-phase root logic
6. Populate `EdgeRef` with real graph data
7. Implement `IntoTransition` trait
8. Create separate query path event stream
9. Move event emission into `RootCursor` (phase 2)
10. Update frontend for all type changes

## Execution Steps

### Phase 1: SearchNode Methods + Remove PoppedNode

**Goal:** Eliminate `PoppedNode`, add convenience methods to `SearchNode`.

- [ ] 1.1: Add `SearchNode::is_parent() -> bool` method
- [ ] 1.2: Add `SearchNode::root_token() -> Token` method (delegates to `root_parent()`)
- [ ] 1.3: Remove `PoppedNode` struct from `iterator.rs`
- [ ] 1.4: Remove `pop_node()` and `process_node()` from `SearchIterator`
- [ ] 1.5: Update `pop_and_process_one` to return a richer `BfsStepResult` that includes `node_index` and `is_parent` (derived via the new methods)
- [ ] 1.6: Update `SearchState::next()` to work with `pop_and_process_one`, injecting `VisitParent` event via a pre-hook or by peeking at the queue
- [ ] 1.7: Verify: `cargo test -p context-search`

**Design Note (1.6):** The current flow is: pop → emit VisitParent (if parent) → process. With `pop_and_process_one`, we need either:
- (a) Peek the queue before popping to emit VisitParent, or
- (b) Have `pop_and_process_one` return the `SearchNode` info alongside `BfsStepResult` so the caller can emit VisitParent before acting on the result, or
- (c) Add an optional callback/closure parameter for pre-process events

Recommended: (b) — enrich `BfsStepResult` to carry `node_index: usize`, `is_parent: bool` on all variants. Emit VisitParent based on these fields before handling the result.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| PathNode in Transition breaks many frontend sites | Medium | Medium | TS compiler catches all type errors; ~28 sites identified in advance |
| IntoTransition trait is too complex for edge cases | Medium | Low | Start with CompareInfo/BfsStepResult; add more impls incrementally |
| Real EdgeRef data not always available (parent expansion) | Medium | Low | Fallback to best-effort; parent expansion can use parent_state data |
| Event collector in advance_to_next_match couples algorithm to viz | Low | Medium | Collector is just `Vec<(Transition, String)>` — no viz dependency |
| Separate query path stream doubles event volume | Low | Low | Query events are lightweight; frontend can filter by path_id |
| Two-phase root requires VizPathGraph state machine changes | Low | Medium | Minimal change — single boolean field + 3 transition handlers |
| Phase ordering — later phases depend on earlier ones | High | High | Each phase has its own verification step; don't proceed if tests fail |
